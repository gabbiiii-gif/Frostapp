# Fase 2.4 — Email OTP no primeiro login

**Data:** 2026-05-24
**Roadmap:** segurança auth, fase 2.4
**Depende de:** Fase 2.3 (convite por email, commit `84197dc`), Resend configurado em Supabase SMTP

## Objetivo

Adicionar uma camada extra de verificação no primeiro login bem-sucedido de cada
membro: o usuário recebe um código de 6 dígitos por email e precisa digitá-lo
antes de acessar o dashboard. Após confirmado uma vez, próximos logins não
pedem mais OTP.

O recurso é **opt-in por empresa** (toggle em Settings). Empresas que não
ligam o toggle continuam com o login atual (senha + 2FA opcional). Master tier
não é afetado.

## Modelo de dados

### Coluna nova em `companies`

```sql
ALTER TABLE companies
  ADD COLUMN require_first_login_otp boolean NOT NULL DEFAULT false;
```

### Coluna nova em `company_members`

```sql
ALTER TABLE company_members
  ADD COLUMN first_login_otp_done boolean NOT NULL DEFAULT false;
```

Marcada `true` quando o membro confirma o OTP do primeiro login. Reset desse
campo por admin é cenário fora do MVP (admin pode usar `update_password` se
quiser forçar reset completo).

### Tabela nova `email_otps`

```sql
CREATE TABLE email_otps (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  company_id uuid NOT NULL REFERENCES companies(id) ON DELETE CASCADE,
  code_hash text NOT NULL,            -- sha256 hex do código (nunca o código raw)
  purpose text NOT NULL,              -- 'first_login' nesta fase; extensível
  expires_at timestamptz NOT NULL,
  attempts int NOT NULL DEFAULT 0,
  consumed_at timestamptz,
  created_at timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX email_otps_active_by_user
  ON email_otps (user_id, purpose)
  WHERE consumed_at IS NULL;
```

RLS:

```sql
ALTER TABLE email_otps ENABLE ROW LEVEL SECURITY;
-- Sem policies: acesso exclusivo via service_role (edge functions).
```

## Edge functions

### `send-email` (helper compartilhado)

Helper reusável que centraliza o envio via Resend. Outras fases (notificação
de OS por email — Fase 2.7) vão consumir o mesmo.

- `verify_jwt: false` (chamado internamente por outras edge functions com
  service_role; não exposto direto ao cliente).
- Input JSON: `{ to: string, subject: string, html: string, text?: string }`
- Lê `RESEND_API_KEY` do `Deno.env`.
- Sender: `noreply@app.frosterp.com.br` (fixo no código por agora; pode virar
  env var depois).
- POST `https://api.resend.com/emails` com headers `Authorization: Bearer ${RESEND_API_KEY}`.
- Retorna `{ ok: true, id }` ou `{ ok: false, error }`.

Secret a configurar no Supabase: `supabase secrets set RESEND_API_KEY=<key>`.

### `first-login-otp-send`

- `verify_jwt: true` — caller precisa estar autenticado (logou com senha).
- Body vazio (deduz user/company do JWT).
- Fluxo:
  1. Pega `auth.uid()`. Carrega `company_members` do user.
  2. Confere `companies.require_first_login_otp = true` E `first_login_otp_done = false`.
     Se não, retorna `{ ok: false, error: 'otp_not_required' }`.
  3. Cooldown: se existe OTP do mesmo user/purpose criado há <60s, retorna
     `{ ok: false, error: 'cooldown', retry_in: N }`.
  4. Invalida OTPs ativos anteriores: `UPDATE email_otps SET consumed_at=now() WHERE ...`.
  5. Gera código 6 dígitos com `crypto.getRandomValues` (range 100000-999999).
  6. `code_hash = sha256(code)`. INSERT em `email_otps` com `expires_at = now() + 10min`.
  7. Chama `send-email` com template pt-BR (subject + html + text).
  8. Retorna `{ ok: true, expires_at }`.

### `first-login-otp-verify`

- `verify_jwt: true`.
- Input: `{ code: string }`.
- Fluxo:
  1. Valida formato (`^\d{6}$`).
  2. Pega `auth.uid()`. Busca OTP ativo (`purpose='first_login'`, `consumed_at IS NULL`,
     `expires_at > now()`) ordenado por `created_at DESC` LIMIT 1.
  3. Se não achar → `{ ok: false, error: 'no_active_otp' }`.
  4. `attempts++`. Se `attempts >= 5`: marca `consumed_at = now()` e retorna
     `{ ok: false, error: 'max_attempts', locked: true }`. Cliente deve forçar
     logout + lockout local 15min.
  5. Compara `sha256(code)` com `code_hash`. Mismatch → `{ ok: false, error: 'invalid_code', attempts_left: N }`.
  6. Match: marca `consumed_at = now()`, `UPDATE company_members SET first_login_otp_done = true`.
  7. Retorna `{ ok: true }`.

## Frontend

### `src/supabase.js`

Novos helpers:

- `sendFirstLoginOTP()` → POST `first-login-otp-send`. Retorna `{ ok, expires_at?, error? }`.
- `verifyFirstLoginOTP(code)` → POST `first-login-otp-verify`. Retorna `{ ok, error?, attempts_left?, locked? }`.

### `src/App.jsx`

**LoginScreen handleSubmit:**

Após `signInWithFallback` OK e `_afterAuth` carregar member, checa:

```jsx
const company = DB.get("erp:company:" + member.company_id);
if (company?.requireFirstLoginOTP && !member.first_login_otp_done) {
  await sendFirstLoginOTP();      // dispara email
  setPendingFirstOTP({ member, email: normalizedEmail });
  setLoading(false);
  return;
}
// segue fluxo normal de login
```

**Novo componente `FirstLoginOTPDialog`:**

- Modal centralizada.
- 6 inputs numéricos (ou um único input com mask).
- Botão "Validar" → `verifyFirstLoginOTP(code)`.
- Sucesso → fecha modal, chama `onComplete(member)` que segue pra `onLogin`.
- Erro → mostra tentativas restantes.
- `locked` → mostra mensagem de bloqueio + forçar `signOutSupabase()` + voltar pra LoginScreen.
- Link "Reenviar código" → desabilitado por 60s (countdown), depois chama `sendFirstLoginOTP` de novo.
- Mostra `expires_at` em countdown (10min).

## Master tier

Master não passa por `company_members`. Login do master segue direto, sem OTP.
Razão: master é usado raramente e tem fluxo separado (`MasterLoginScreen`).
Adicionar OTP para master fica como item futuro se virar requisito.

## Rate limiting & lockout

- Cooldown reenvio: 60s entre solicitações de novo código.
- Validade OTP: 10 minutos.
- Tentativas: 5 erradas → OTP consumido + frontend bloqueia conta 15min via
  `frost_login_attempts` (reusa lockout endurecido da Fase 2.1).
