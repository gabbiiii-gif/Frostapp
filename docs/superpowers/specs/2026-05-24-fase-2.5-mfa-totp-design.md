# Fase 2.5 — 2FA TOTP via Supabase MFA built-in

**Data:** 2026-05-24
**Roadmap:** segurança auth, fase 2.5
**Depende de:** Fase 2.4 (estrutura CompanySecurityPanel + companies flag), 2FA custom legacy existente (será descontinuado em runtime)

## Objetivo

Migrar 2FA TOTP do generateTotpSecret/verifyTotp custom (factors em
`erp:user:*` local) para `supabase.auth.mfa.*` built-in. Resultados:
- Factors no `auth.users` → cross-device automático.
- Rate-limit e audit trail server-side.
- Admin pode resetar 2FA de outro usuário (recurso "perdi celular").
- Toggle por empresa pra exigir 2FA obrigatório.
- Sem backup codes (não suportado pelo built-in; reset via admin é o caminho).

## Schema

```sql
ALTER TABLE companies
  ADD COLUMN require_mfa boolean NOT NULL DEFAULT false;
```

## Edge function

### `admin-remove-user-mfa`

- `verify_jwt: true`
- Body: `{ user_id }`
- Confere caller é `admin`/`gerente` da mesma `company_id` do target via
  `company_members`. `is_super_admin` ignora a checagem de mesma company.
- Lista factors via `admin.auth.admin.mfa.listFactors({ userId })` e deleta
  cada um via `admin.auth.admin.mfa.deleteFactor`.
- Retorna `{ ok: true, removed: N }`.

## Helpers em `src/supabase.js`

- `listMfaFactors()` — wrapper de `auth.mfa.listFactors()`.
- `enrollMfaTotp(friendlyName)` — chama `auth.mfa.enroll({ factorType: 'totp', friendlyName })`, retorna `{ factorId, qr, secret, uri }`.
- `challengeMfa(factorId)` — dispara desafio.
- `verifyMfaChallenge(factorId, challengeId, code)` — completa verify.
- `challengeAndVerifyMfa(factorId, code)` — atalho enroll (junta challenge + verify).
- `unenrollMfa(factorId)` — usuário desativa próprio factor.
- `adminRemoveUserMfa(targetUserId)` — chama edge function.

`_afterAuth` carrega `companies.require_mfa` em `member.company_require_mfa`.

## Componentes

### `TwoFactorAuthPanel` (refatorado)

- Lista factors no mount via `listMfaFactors`.
- `verifiedTotpFactor` = factor TOTP com `status === "verified"`.
- Enroll: limpa factors `unverified` órfãos → `enrollMfaTotp` → mostra QR + secret + input código → `challengeAndVerifyMfa` → limpa campos legacy + reload.
- Disable: `unenrollMfa(verifiedTotpFactor.id)` → limpa campos legacy + reload.
- Badge "Reenrolar" quando `hadLegacy2FA && !enabled`.
- Sem backup codes (nota informando "peça admin pra resetar se perder celular").

### `CompanySecurityPanel` (extensão)

Agora tem dois toggles:
1. `require_first_login_otp` (Fase 2.4).
2. `require_mfa` (Fase 2.5).

### `LoginScreen` (refatorado)

Após `signInWithFallback` OK + `_afterAuth` carregar member + passar pelo gate
da Fase 2.4 (OTP por email):

1. `listMfaFactors()` → `verifiedTotp = totp.find(status === 'verified')`.
2. Se `verifiedTotp` existe:
   - `challengeMfa` → `setPendingMfaChallenge({ user, factorId, challengeId })`.
   - Tela "Verificação em 2 etapas" → `handleVerifyMfaChallenge` → `verifyMfaChallenge` → `onLogin`.
   - Em caso de código errado, gera novo challenge automaticamente.
3. Se `require_mfa` ON e sem `verifiedTotp`:
   - Limpa factors `unverified` órfãos.
   - `enrollMfaTotp` → `setPendingMfaEnroll({ user, factorId, qr, secret, reason })`.
   - Tela "2FA obrigatório" → `handleConfirmMfaEnroll` → `challengeAndVerifyMfa` → cleanup legacy local + `onLogin`.
4. Se `hadLegacy` sem `require_mfa` e sem factor → cleanup legacy local + segue login normal.

Cancelar (`handleCancelMfa`) → `signOutSupabase` + volta ao login.

### `UserManagement` (extensão)

Botão "Resetar 2FA" por linha (não aparece pra própria linha). Confirm via
`window.confirm` → `adminRemoveUserMfa(u.authUserId)` → toast.

## Migração silenciosa

- Online sem `require_mfa`: detecta `twoFactorEnabled` legacy, limpa campos
  (`erp:user:*`), login segue. Panel mostra badge "Reenrolar".
- Online com `require_mfa`: enroll forçado inline durante login.
- Offline (fallback local em `signInWithFallback`): aceita login sem 2FA
  (degradação aceita — Supabase MFA exige conexão).
